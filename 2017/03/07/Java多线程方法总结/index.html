2.wait、notify、notifyAll和sleep方法

  sleep是线程类Thread的静态方法，导致此线程暂停执行指定时间，给执行机会给其他线程，但是监控状态依然保持，到时后会自动恢复。调用sleep不会释放对象锁。
  wait是Object类的方法，对此对象调用wait方法导致本线程放弃对象锁，进入等待此对象的等待锁定池，只有针对此对象发出notify方法或notifyAll后本线程才进入对象锁定池准备获得对象锁进入运行状态。
这两个方法的区别如下：

  1. sleep是Thread类的静态方法，wait是Object类的非静态方法。
  2. sleep不会使线程释放对象锁，wait会使线程释放锁。
  3. wait，notify和notifyAll只能在同步控制方法或者同步控制块里面使用，而sleep不需要

  当调用sleep方法后，线程会立即进入休眠状态，但线程并不会释放他所持有的所有对象锁。过了指定休眠时间后，线程会再次进入可运行状态，继续争夺cpu资源，当获得cpu资源后，线程继续执行。
  wait属于Object的成员方法，一旦一个对象调用了wait方法，必须要采用notify()和notifyAll()方法唤醒该进程。如果线程拥有某个或某些对象的同步锁，那么在调用了wait()后，这个线程就会释放它持有的所有同步资源，而不限于这个被调用了wait()方法的对象。
wait和notify使用示例：

当线程B调用obj.notify()后，线程A被唤醒，此时线程A会被置于对象锁等待池中等待obj锁。当线程B调用obj.notify()并不会释放obj锁，而是等到obj同步块执行完毕后才释放，此时线程A会继续争夺obj锁，得到锁后继续执行。

需要注意的概念是：

  1. 调用obj的wait()， notify()方法前，必须获得obj锁，也就是必须写在synchronized(obj) {……}代码段内（或synchronized方法中）。
  2. 调用obj.wait()后，线程A就释放了obj的锁，否则线程B无法获得obj锁，也就无法在synchronized(obj) {……} 代码段内唤醒A。
  3. 当obj.wait()方法返回后，线程A需要再次获得obj锁，才能继续执行。
  4. 如果线程A，C，D都已经调用obj.wait()，则线程2调用obj.notify()只能唤醒A，C，D中的一个（具体哪一个由JVM决定）。此时如果调用obj.notifyAll()则能全部唤醒线程A，C，D。
  5. 当线程B调用obj.notify/notifyAll的时候，线程B正持有obj锁，因此，A，C，D虽被唤醒，但是仍无法获得obj锁。直到线程B退出obj的synchronized块，释放obj锁后，A，C，D中的一个才有机会获得锁继续执行。

为什么wait()，notify()方法要和synchronized一起使用？
因为wait()方法是通知当前线程等待并释放对象锁，notify()方法是通知等待此对象锁的线程重新获得对象锁，然而，如果没有获得对象锁，wait方法和notify方法都是没有意义的，即必须先获得对象锁，才能对对象锁进行操作，于是，才必须把notify和wait方法写到synchronized方法或是synchronized代码块中了。

yield方法

  yield是Thread类的静态方法，当调用此方法时，会使当前线程从运行状态变为可运行状态。也就是说，当前线程会让出cpu资源，和其他可运行状态的线程一起重新争夺cpu资源。如果当前线程在yield之后又立即抢到了cpu资源，那么线程还是可以继续执行。这里不再代码演示。
